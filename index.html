Tool vs. Real SDK Rendering — Beacon Firing Logic

Impression Beacon

Real HyBid SDK: Impression beacons are fired using IntersectionObserver — the ad must be ≥50% visible for at least 1 continuous second before any impression pixel fires. This matches the IAB OMID standard. For video, the VAST <Impression> URLs fire on the first play event.

This tool:

  • Types 1/2/3 (banner): The beacon interceptor captures Image.src, fetch, and XHR calls fired from inside the iframe — so any impression pixels the creative fires itself are logged. However, the tool does not add an external viewability gate for these types. If the creative fires its impression pixel unconditionally on load (common), it will fire immediately rather than after a 1s viewability threshold.
  • Type 4 (MAX_ADM): The OMID logic is replicated — ext.imptrackers are held back and only fired once the ad frame is ≥50% visible for 1 continuous second, matching real SDK behaviour.
  • VAST video (all types): VAST <Impression> URLs fire on the first play event — identical to the SDK.

Click Beacon

What the tool does

The ad renders inside an iframe.srcdoc in a browser tab. For Types 1/2/3, the Verve wrapper then creates a second nested iframe (#ctr) inside that — so there are two iframe layers:

Browser tab
  └── iframe#ad-frame  (srcdoc, tool-injected)
        └── iframe#ctr  (Verve wrapper JS)
              └── actual DSP creative

A real mobile SDK (HyBid)

The SDK renders the creative inside a WebView. The Verve wrapper also creates a nested #ctr iframe inside the WebView:

Native app
  └── WKWebView / WebView  (SDK-managed)
        └── iframe#ctr  (Verve wrapper JS)
              └── actual DSP creative

Does the extra outer iframe affect click beacon firing?

For the Verve wrapper script (Types 2/3) — functionally no. The click beacons in the wrapper (CB[]) are fired from inside the #ctr iframe via a click event listener using new Image().src. This behaviour is identical whether the wrapper runs in a srcdoc iframe or a native WebView.

For DSP creatives that use window.top or parent traversal — there is a difference. Some creatives navigate clicks using window.top.location or getWindowTop().open(...). In a real SDK WebView, window.top is still the WebView. In the tool, it escapes to the browser tab — click navigation opens a new browser tab rather than a deep-link or App Store redirect (expected for testing), but click pixel firing via Image.src is unaffected.

For Types 2/3/4 (post-ad-server responses), click beacon behaviour is structurally identical to a real SDK WebView.
