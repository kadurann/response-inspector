Tool vs. Real SDK Rendering — Beacon Firing Logic

Ad Rendering

Real HyBid SDK: Banner and native ads render inside a WKWebView / WebView managed by the native app. The SDK controls the lifecycle — viewability, impression timing, click interception — at the native layer, independent of JavaScript event propagation.

This tool:

  • Types 1/2/3 (banner/wrapper): The adm renders inside a tool-injected iframe.srcdoc. The Verve wrapper then creates a second nested #ctr iframe inside it — so there are two layers vs. the SDK's one WebView + one iframe. The wrapper JS, MRAID stub, and macro resolution run identically to the SDK. Click beacons in CB[] are captured via a capture-phase safety net on the outer frame. ✓ Creative execution and beacon firing behaviour match the real SDK.
  • Native (oRTB native adm): The real SDK passes raw asset objects to the publisher's own native view (UIView / View). The tool renders a reference in-feed card inside an iframe to approximate this, with a simulated feed context (fake posts, app bar) for visual realism. ⚠ Layout is the tool's own reference implementation, not the publisher's actual view.
  • Type 4 (MAX_ADM): remoteconfigs (skip offset, audio state, endcard behaviour, icon size) are fully parsed and applied during rendering. ✓ Config application matches real SDK behaviour.
  • VAST video: Rendered in a native <video> element with full wrapper chain resolution (up to 5 hops), quartile tracking, companion endcard, and custom endcard. ✓ Structurally identical to the SDK's video player.

Impression Beacon

Real HyBid SDK: Impression beacons are fired using IntersectionObserver — the ad must be ≥50% visible for at least 1 continuous second before any impression pixel fires. This matches the IAB OMID standard. For video, the VAST <Impression> URLs fire on the first play event.

This tool:

  • Types 1/2/3 (banner): The beacon interceptor captures Image.src, fetch, and XHR calls fired from inside the iframe. No external viewability gate is added — if the creative fires its impression pixel on load, it fires immediately rather than after a 1s threshold. ⚠ Timing differs from SDK; pixel URLs and firing mechanics are identical.
  • Native (oRTB native): imptrackers[] and eventtrackers[] (event=1, method=1) are fired immediately when the card is built — equivalent to after-render in a WebView. No viewability gate is applied. ⚠ Fires on render rather than on viewability — earlier than the real SDK in practice.
  • Type 4 (MAX_ADM): OMID logic is fully replicated — ext.imptrackers are held back until the ad frame is ≥50% visible for 1 continuous second. ✓ Matches real SDK behaviour exactly.
  • VAST video (all types): VAST <Impression> URLs fire on the first play event. ✓ Identical to the SDK.

Click Beacon

Types 1/2/3 — banner/wrapper: The ad renders inside an iframe.srcdoc, within which the Verve wrapper creates a nested #ctr iframe:

Browser tab
  └── iframe#ad-frame  (srcdoc, tool-injected)
        └── iframe#ctr  (Verve wrapper JS)
              └── actual DSP creative

vs. real SDK:

Native app
  └── WKWebView / WebView  (SDK-managed)
        └── iframe#ctr  (Verve wrapper JS)
              └── actual DSP creative

The extra outer iframe is functionally invisible to the CB[] click beacons — a capture-phase listener on the outer frame reads the wrapper's CB[] array directly (same-origin) and fires them even when the inner creative suppresses event propagation. Click navigation (window.top.open()) opens a browser tab instead of a deep-link — expected for testing. ✓ Click pixel URLs and firing mechanics match the real SDK for Types 2/3.

Native (oRTB native): link.clicktrackers[] fire on card click and are reported to the beacon panel. The real SDK fires these at the native tap handler level; the tool fires them from a JS click listener on the feed card. ✓ Behaviourally equivalent for beacon testing purposes.

For Types 2/3/4 (post-ad-server responses), click beacon behaviour is structurally identical to a real SDK WebView.
